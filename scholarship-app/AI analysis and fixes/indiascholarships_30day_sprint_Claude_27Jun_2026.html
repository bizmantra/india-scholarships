2 things the architecture docs flag that the traffic data confirms
Pipeline bottleneck

Content update is 5 manual steps

Google Sheets → SQLite → Gemini enrichment → WP All Import → Vercel rebuild. Every CTR fix goes through this. Automating it with GitHub Actions is the single highest leverage infra task.

Brand trust risk

Deadlines are calculated at build-time

A page can show "Apply Now" for a scheme whose deadline passed weeks ago. For a site built on verified accuracy, this is a silent trust-killer. Needs a client-side JS fix.

FAQPage schema — implementation spec

+

Your faq_json field already stores exactly 3 Q&A pairs per scholarship. The implementation is a direct mapping into FAQPage JSON-LD. Replace the existing Article script block in app/scholarships/[slug]/page.tsx:

// Replace this in generateMetadata or in the page component const faqSchema = { "@context": "https://schema.org", "@type": "FAQPage", "mainEntity": JSON.parse(scholarship.faq_json).map(faq => ({ "@type": "Question", "name": faq.question, "acceptedAnswer": { "@type": "Answer", "text": faq.answer } })) };

Add this alongside (not replacing) the Article schema — Google supports multiple schema types on the same page. Validate with Google's Rich Results Test before deploying. After deployment, check GSC's Search Appearance tab — FAQ rich results typically appear in 2–4 weeks if the content passes quality checks.

FAQs must be genuine Q&As, not marketing copy. Google rejects FAQ schema where answers are vague or promotional. Your Gemini-enriched FAQs should already meet this bar since they're pulled from official sources via grounding.

The subpage architecture — what the data tells you vs. what the code promises

+

Your architecture docs describe a full subpage system: /eligibility, /income-limit, /documents-required, /last-date, /selection-process, /apply-online, /renewal-process. The code supports all of these. But GSC shows only 28 subpages indexed, almost all for PM Yashasvi only.

The reason is almost certainly that these pages aren't being generated at build time for most scholarships — either the database fields required (step_guide, docs_needed, deadline) aren't populated for most records, or the generateStaticParams() call is filtering them out.

The proof that subpages work is in your own data. Look at the PM Yashasvi apply-online subpage: it has only 362 impressions but converts at 1.93% CTR — far higher than the main page's 4.14% CTR at much higher volume. And the "pm yashasvi scholarship apply online 2026" query has 24,431 impressions at 17.57% CTR — that's students who found a dedicated apply-online subpage and clicked at 4× the rate of the main page.

Before generating 120 subpages, run content-quality-audit.js and confirm that step_guide and docs_needed are populated for your target 20 scholarships. A subpage with empty fields is worse than no subpage — it's a thin content penalty risk.

GitHub Actions cron — the pipeline automation spec

+

This is the highest leverage infrastructure task in the sprint. Without it, every content update is a manual 5-step process. With it, the pipeline runs automatically and you only need to human-review the Google Sheets output before it goes live.

# .github/workflows/weekly-enrichment.yml name: Weekly Content Enrichment on: schedule: - cron: '0 0 * * 0' # Every Sunday midnight UTC workflow_dispatch: # Also allow manual trigger jobs: enrich: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Run quality audit run: node scripts/content-quality-audit.js - name: Run Gemini enrichment run: node scripts/enrich-all-low-ctr-gemini.js env: GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }} - name: Commit updated database run: | git config user.email "bot@indiascholarships.in" git add data/scholarships.db git commit -m "chore: weekly content enrichment $(date +%Y-%m-%d)" git push - name: Trigger Vercel redeploy run: curl -X POST "${{ secrets.VERCEL_DEPLOY_HOOK }}"
Add a Slack or email notification step so you get alerted when the cron runs. Review the Google Sheets output before the WordPress sync step if you want a human checkpoint — you can make that step manual-trigger only.

Monetisation — what your current traffic already supports

+

At 44,000+ clicks/month and growing, you're crossing the threshold where monetisation becomes meaningful. Three paths that fit your audience and architecture:

1. Google AdSense (immediate, low effort). Scholarship sites in India run at ₹80–150 RPM (revenue per 1,000 pageviews) for AdSense. At 1,00,000 monthly sessions, that's ₹8,000–15,000/month. Not life-changing but zero work once set up. Place ads below the fold on detail pages — never between the eligibility checker and the results.

2. Education loan affiliate (medium effort, higher value). Students who find a scholarship often need a backup plan — an education loan. Partners like Credila, Avanse, and HDFC Credila pay ₹2,000–5,000 per qualified lead. A single "Didn't qualify? Here's how to fund your education" CTA on the eligibility checker results page could generate meaningful revenue. Your architecture docs mention financial products as a high-CPM niche — this is how you access it without building a new site.

3. Email list monetisation (high effort, highest long-term value). Once Task 11 (email capture on eligibility checker) builds a list, you have a segmented database of students by state, caste, income bracket, and education level. Coaching institutes, laptop brands, and edtech platforms targeting this cohort will pay for email placements. At 500 subscribers/month, you have a meaningful list within 6 months.

Don't run ads on the eligibility checker or between application steps. The checker is your trust-building and differentiation asset — ad clutter there directly undermines the brand trust metric that the entire architecture is built around.
